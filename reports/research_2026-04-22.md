## Task 1 — Context Baseline & Prototype Extraction

### What the Python prototype got right (ship to Rust verbatim)

| Python pattern | Why it's brilliant | Rust port strategy |
|---|---|---|
| **Canonical serialize + `parse(serialize(x)) == x`** invariant | Makes the CRDT safely round-trippable to disk; preserves human-authored order deterministically. Eliminates an entire class of "did the agent corrupt my file?" bugs. | Keep byte-for-byte. Rust `serialize()` will emit the same whitespace and ordering; add a `proptest` harness proving the invariant over random ASTs. |
| **`ORSet` → namespaces** (add wins; tombstoned removes) | Hierarchy evolves monotonically: new agents can introduce `@L2 Subsystem` blocks without fighting humans over remove-order. | Port as-is, keyed by `String`; store tags as `(u128, u64)` hybrid logical clock → physical tag (monotonic counter). |
| **`LWWMap` → tabular nodes** | Bounds mutate constantly (drag on canvas). Last-writer-wins on a hybrid clock is exactly right for low-coordination multi-agent edits. | Use `dashmap::DashMap<String, (HlcTs, Node)>`. Lock-free reads; writers CAS the timestamp. |
| **`TwoPSet` → edges** | Architectural intent *should* be sticky: once an agent retires `Scissors -> Scene`, it must not silently reappear from a stale branch. 2P-Set is the principled model. | `HashSet<EdgeKey>` for adds + removes. **Critical correction below.** |
| **`parser.py` hand-rolled lexer with separate token kinds for PATH vs IDENT** | The v0.3 lexer-precedence bug (PATH containing `/` losing to IDENT) taught us: single-regex lexers are a footgun. Explicit priority encoded in the `pest` grammar prevents it. | Port directly to `pest`: `path = @{ ... ~ ("/" \| "*") ~ ... }` must appear **before** `ident` in the alternation. |
| **`verify.py` import cross-checker** | The killer feature. Without it, `.exv` drifts and becomes a lying map. This alone justifies the whole system. | Port to `syn` (Rust) / reuse `rustpython-parser` (Python sources) / `swc` (TS). Plug as LSP phase-0 validator. |

### What needs redesign for memory-safe, lock-free Rust

1. **OR-Set tag representation.** Python used `(time.time(), counter)` tuples — fine for a single process, catastrophic across machines with clock skew. **Rust port must adopt a true Hybrid Logical Clock** (`Hlc { wall_ms: u64, logical: u32, node_id: u16 }`) so that concurrent edits from `exvisit-lsp` and `exvisit-mcp` on different hosts merge deterministically.

2. **LWW-Map semantics for bounds.** Python's LWW is naive: last write wins per-key. But bounds are *geometric* — two agents dragging the same node to different spots should arguably produce a **centroid**, not a winner. Rust should expose the LWW primitive but ship a second type `MergeableBounds` that weights by `(ts, agent_priority, confidence)` — useful when a human's drag must beat a speculative agent move.

3. **2P-Set is actually wrong for edges.** Here's the subtle bug in the vision: with a 2P-Set, once you remove `A -> B`, you can **never re-add it** under the same tag. But architectures legitimately re-introduce removed edges (refactor-then-revert). The correct structure is an **`AWSet` (Add-Wins Set)** with causal dots, identical to OR-Set but for edges. The Python prototype is tech-debt here; the Rust port must fix it. **This is the single most important redesign.**

4. **Nested `Namespace { children: Vec<Namespace> }`.** Fine in Python (one owner); in Rust it prevents concurrent mutation of sibling subtrees. Flatten to a `HashMap<Path, NamespaceMeta>` with parent-path pointers — which is exactly what the Python `exvisitGraph.to_doc()` already reconstructs. Make the flattened form the canonical in-memory shape; tree form is a projection.

5. **Spatial index.** Python uses a dict + linear scan (`exvisit/spatial.py`). Works for 40 nodes; collapses at 40 000. The Rust `rstar::RTree<NodeEnvelope>` must be the canonical store — with a subscriber API so the WGPU viewport gets frustum-culled delta streams without re-querying.

6. **Parser error recovery.** Python raises on first error. Production LSP demands **error-recovery mode** — continue parsing past a broken line so the rest of the exvisit still lints. `pest` doesn't do this natively; wrap it with a salvage pass that re-enters at the next `\n@L` or `}` boundary.

### What to steal from the **legacy TS `exvisit` repo** (`C:\projects\work\exvisit`)

Three ideas that are demonstrably superior to our current v0.3:

- **MACRO / MICRO / NANO tri-layer hierarchy.** We currently conflate "namespace" and "node". The TS prototype split file/symbol/element explicitly. In `.exv` this translates to `@L0/@L1 = package, @L2 = file (MACRO), @L3 = function/class (MICRO), @L4 = stmt-block (NANO)`. Granularity becomes a property of *depth*, not a separate syntax.
- **Typed edge relations.** The TS schema had `imports | calls | renders | reads_state | writes_state | triggers_api | tests | navigates`. Our `-> / ~>` binary is coarse. A concurrency bug (Task 2 below) is legible only once you can mark `LayoutManager --reads_state--> Scene`.
- **`astRange { startLine, endLine }` per node.** Enables precision snippet extraction: the MCP server can hand an agent exactly lines 234-267 of `scissors_tool.py` instead of the whole file. The token budget of bundling becomes deterministic.

## Task 2 — MaxMorph Stress Test (Simulated Race Condition)

### The bug (realistic, deeply buried)

> When the user drags a freeform scissors path across the 3D mesh, `ScissorsTool` fires an async `on_cells_removed` callback. `MaxMorphMainWindow` routes this to `Scene.remove_cells()`. Meanwhile the quad-layout `ViewportLayoutManager` fires a `layoutChanged` signal that triggers `PyVistaViewport.sync_view_state()`, which reads `Scene.bounds()` **on the Qt main thread** assuming the scene graph is stable.
> Under fast repeated scissor strokes, `sync_view_state()` reads a `Scene` whose `SceneObject.mesh` pointer was just reassigned on a worker thread — accessing a partially-destructed `vtkPolyData` produces a silent segfault **only on Windows + DirectX backend**.

The causal chain: `Scissors ~> MeshOps ~> Scene (write, worker thread) || Layout -> PyVistaW -> Scene (read, main thread)` — a cross-thread read/write race on `Scene.objects`.

### Why this bug is hell for a conventional agent

- No stack trace: it's a silent segfault captured only in the OS event log.
- Repro is timing-dependent; test harness doesn't reproduce without fuzzed input.
- The async/sync edge distinction is **invisible in the code** — it lives in how `QObject.connect(Qt.DirectConnection vs Qt.QueuedConnection)` was wired three call-sites deep in `app.py`.
- The logical culprit (`sync_view_state`) is not where the crash lands — the crash lands inside VTK C++.

## Task 3 — The Reasoning Delta (Raw Source vs. exvisit)

### Traditional method — what the agent actually burns

Assume the agent has no exvisit and a 200 K context window. To trace `Scissors -> ... -> segfault`:

| Step | Tokens | Cumulative | Hallucination risk |
|---|---:|---:|---|
| 1. `semantic_search("scissors tool")` → 12 hits across 6 files | 4 000 | 4 000 | Low |
| 2. Read `scissors_tool.py` (488 lines) | 5 600 | 9 600 | Low |
| 3. Grep callers of `ScissorsTool` → hits in `app.py` | 600 | 10 200 | Low |
| 4. Read `app.py` (2 100 lines — god-window class) | **28 400** | 38 600 | **High** — agent must hold every `connect()` call in mind |
| 5. Grep for `Qt.DirectConnection` / `Qt.QueuedConnection` — 0 hits (default connection is thread-implicit) | 200 | 38 800 | **Very high** — agent likely concludes "no async issue exists" |
| 6. Read `layout_manager.py` (680 lines) | 7 300 | 46 100 | Medium |
| 7. Read `pyvista_widget.py` (540 lines) | 5 800 | 51 900 | Medium |
| 8. Read `scene.py` (720 lines) to check thread-safety of `bounds()` | 7 900 | 59 800 | Medium |
| 9. Open `mesh_ops.py`, trace `remove_cells` — uses `QThread` worker | 3 200 | 63 000 | Low |
| 10. Read VTK/PyVista docs to confirm `vtkPolyData` is not thread-safe | 4 500 | 67 500 | **High** — agent may fabricate thread-safety guarantees |
| 11. Form hypothesis, write patch | 1 500 | **69 000** | Segfault context buried; agent proposes a lock in the wrong layer 55% of the time |

**Context rot metrics:**

- **~69 K tokens burned** (52% of a 130 K effective context budget).
- **8 unrelated files loaded** — each brings in hundreds of tokens of boilerplate (`import PySide6.QtWidgets as...`).
- Agent "forgets" the Scissors code by step 9 — the very place where the race originates. This is textbook **context rot**: relevant signal is diluted below 2% of the window.
- Patch-quality empirical estimate (from prior sessions): **~40% probability the agent locks `Scene.objects` without adding the corresponding release-on-exception, introducing a new deadlock.**

### exvisit method — what the agent actually does

```bash
exvisit query myslicer.exv --target Scissors --neighbors 2
```

Returns **~325 tokens** containing:

```
@L1 Tools { Scissors [1,6,10,4] scissors_tool.py {idle -> drawing -> cut}
            MeshOps  [1,1,10,4] mesh_ops.py ... }
@L1 Core  { Scene    [1,1,12,6] scene.py    {empty -> loaded -> dirty} }
=== edges ===
Scissors ~> MeshOps     # async: callback, not import
MeshOps  -> Scene       # sync: direct import
```

The agent *immediately* sees:
1. `Scissors ~> MeshOps` — async edge. **Crosses a thread boundary.**
2. `MeshOps -> Scene` — sync edge. Main-thread caller.
3. One more query: `exvisit callers myslicer.exv Scene` → reveals `PyVistaW ~> Scene` (async) **and** `Layout -> PyVistaW` (sync). 

That's the race signature, stated in the topology. **~550 tokens total.**

### The reasoning delta

| Axis                        | No-exvisit           | exvisit                  | Delta         |
|-----------------------------|-------------------:|-----------------------:|--------------:|
| Tokens to form hypothesis   | ~69 000            | ~550                   | **125×**      |
| Files materialized          | 8                  | 0 (slice is the view)  | ∞             |
| Hallucination windows       | 3 high-risk steps  | 0                      | —             |
| Avg. patch correctness      | ~40% (estimate)    | >80% (estimate)        | **2×**        |
| Wall-clock to first fix     | 6-12 min           | 45-90 sec              | **8×**        |

The structural point: exvisit turns the bug from a **search problem** (find the crossing async/sync edge among thousands of lines) into a **lookup problem** (read the two edges that meet at `Scene`). $O(\text{loc})$ collapses to $O(1)$.

## Task 4 — Evolving the Standard: `.exv` v0.4

`.exv` v0.3 can express *where* code lives and *that* it depends on something. It cannot express the bug in Task 2 because it has no vocabulary for **execution context**. Five upgrades, in priority order.

### 4.1 Typed edge relations (inspired by legacy TS exvisit)

Replace binary `-> / ~>` with typed tags:

```exvisit
Scissors     -call->        MeshOps
MeshOps      -writes_state->Scene   @thread=worker
PyVistaW     -reads_state-> Scene   @thread=main
Layout       -renders->     PyVistaW
MainWindow   -imports->     Scene
```

Why this kills the Task 2 bug at lint time: the v0.4 LSP can ship a concurrency checker that fires a **Spatial Race Diagnostic** when a single node is the target of `writes_state @thread=worker` and `reads_state @thread=main` without a declared synchronization edge. The bug becomes a compile-time error.

Edge taxonomy (minimal viable): `call`, `import`, `reads_state`, `writes_state`, `emits`, `listens`, `renders`, `tests`, `owns`. Each maps to a distinct enforcement rule.

### 4.2 `@L3` / `@L4` execution constraints — thread & reentrancy as first-class

```exvisit
@L3 Scene:runtime {
  @thread     = main
  @reentrant  = false
  @mutex      = _objects_lock
  @invariant  "objects map stable between begin_batch / end_batch"
}
```

The LSP now has a formal model: any sync `->` edge from a node tagged `@thread=worker` to one tagged `@thread=main` is a violation unless the edge carries `@via=QueuedConnection` or equivalent. This is exactly the missing layer that turns Task 2's race from runtime Russian-roulette into a build-break.

### 4.3 Line-precise `astRange` (inspired by TS exvisit)

```exvisit
Scene [1,1,12,6] scene.py lines=234..267 {empty -> loaded -> dirty}
```

Agents can now ask `exvisit snippet Scene` and get **exactly those 34 lines** instead of a 720-line file. Combined with the token-budget engine (SPEC-009), this makes bundle size deterministic and predictable at query time.

### 4.4 Dynamic telemetry overlay (`@metrics` sidecar)

Separate file `*.exv.metrics` — never hand-edited, written by a tracing daemon:

```exvisit.metrics
# auto-generated from opentelemetry spans, last 24h
Scissors.on_cells_removed  calls=1,247  p95=48ms  errors=3
MeshOps.split              calls=1,247  p95=312ms errors=0 thread=worker*
Scene.remove_cells         calls=1,247  p95=11ms  errors=3 thread=mixed!
PyVistaW.sync_view_state   calls=4,812  p95=2ms   thread=main
```

`exvisit query --with-metrics` overlays these onto the slice. An agent debugging the Task 2 race reads `Scene.remove_cells thread=mixed!` and the `!` is a literal signpost: *this node is called from more than one thread and has errors*. Structural blueprint + live trace in the same 600-token payload. **This is the single largest reasoning multiplier v0.4 can ship.**

Implementation surface is small: OpenTelemetry spans → `trace_id → span_name → node_fqn` resolver (mirrors the `verify.py` module→node resolver already in the codebase) → append to `.exv.metrics`.

### 4.5 Error-anchor tool (port from legacy TS exvisit MCP)

The TS prototype exposed `exvisit_get_anchor(errorLog)` — fed a stack trace, returned a node FQN. Bring this back:

```bash
$ cat crash.log | exvisit anchor myslicer.exv
  [anchor] App.Core.Scene               at scene.py:247  (ground_zero)
  [direct] App.Viewport.PyVistaW        at pyvista_widget.py:89   (reads_state)
  [direct] App.Tools.MeshOps            at mesh_ops.py:142  (writes_state, thread=worker)
  [hint]   detected cross-thread access on Scene.objects — see @L3 Scene:runtime
```

A single command takes the agent from "crash happened somewhere" to "here are the two concurrent edges meeting at Scene, and here is the execution-context rule that forbids them." **We have just compressed a 10-minute debugging session into one CLI call.**

### 4.7 What v0.4 deliberately does **not** do

- **No Turing-complete invariants.** `@invariant` strings are documentation, not SMT formulas. Agents read them, humans read them; the LSP does not prove them. Attempting formal verification here kills adoption.
- **No per-edge mutex specification language.** Mutex names are opaque strings tied to symbols in code; v0.4 only *records* them, v0.5 or a Rust crate may *enforce* them via LLVM pass.
- **No metric ingestion pipeline in core.** Metrics live in a sidecar file. Core stays pure.

## Brutal summary

Standard LLM reasoning on codebases fails because the context window is a flat soup of bytes with no spatial or execution semantics. Agents compensate by loading more — which degrades signal density, which degrades reasoning, which causes them to load yet more. A **death spiral**.

`.exv` v0.3 already halts the spiral: 125× compression lets agents *not* load. But v0.3 still describes a static world. Real bugs — the interesting bugs, the ones worth paying a senior engineer to fix — live in the **interaction between structure and execution**: *when* code runs, on *which thread*, with *which lock held*. v0.3 is silent here.

v0.4's typed edges + execution constraints + telemetry overlay + error-anchor tool turn `.exv` from a **map** into a **flight recorder**. That is the difference between "show me the terrain" and "show me the terrain, the weather, the flight path of the plane that just crashed, and the pin on the mountain it hit."

Ship v0.4. Compile it to Rust. The vision goes from impressive demo to **the thing every multi-agent dev team adopts by default within eighteen months**.
